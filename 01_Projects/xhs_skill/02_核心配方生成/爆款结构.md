
> 适用场景： 小红书 B 端获客、个人品牌建设、行业观点输出
> 
> 核心逻辑： 信任 = 真实场景切片 + 商业冷思考 + 架构师视角

---

## 一、核心原则 (The Constitution)

**1. 拒绝“小编感”，坚持“创始人感”**

- ❌ 绝对禁止：惊叹号连用、震惊体（“天呐！”）、卖课口吻（“建议收藏”）、讨好感（“宝子们”）。
    
- ✅ 必须坚持：平视视角、商业理性、冷峻直接、甚至带一点傲气。
    

**2. 拒绝“搬运”，坚持“切片”**

- ❌ 绝对禁止：复述新闻、解释名词（什么是 Agent）、空对空的理论。
    
- ✅ 必须坚持：我刚经历的、我刚看见的、我刚算过的一笔账。
    

**3. 拒绝“神化 AI”，坚持“架构思维”**

- ❌ 绝对禁止：AI 取代人类、AI 无所不能。
    
- ✅ 必须坚持：人定义野心，AI 负责分母。强调 **Human-in-the-loop (人在回路)**。

## 三、禁用话术清单 (Banned Phrases)

**⚠️ 以下话术带有浓浓的 AI 味，绝对禁止使用：**

1. **"这不是～～～而是～～"排比句**
   - ❌ "这不是效率问题，而是认知问题"
   - ❌ "这不是工具的问题，而是思维的问题"
   - 原因：GPT 最爱用的句式，一眼假

## 二、两种写作模式 (Two Writing Modes)

根据话题深度和目标受众，选择合适的模式：

### 模式A：深度论述模式（推荐用于B端决策者）

**适用场景：** 战略级话题、技术趋势分析、商业观察、需要展示思考深度

**核心结构：** 多维度拆解（6-8个维度） + 连贯段落论述 + 技术/商业类比

**写作要点：**
1. **开篇：** 抛出核心数据/现象 + TLDR 总结（让读者快速判断是否值得读）
2. **主体：** 用编号维度拆解问题（如：1. 成本结构变化、2. 决策链迁移、3. 人机分工...）
3. **每个维度：** 3-5段连贯论述，避免 bullet points 堆砌
4. **类比：** 使用技术类比（信息论、搜索问题）或商业理论（帕金森定律、边际成本）
5. **结尾：** 平衡结论，不是非黑即白

**为什么有效：**
- 展示"架构思维"而非"营销话术"
- 符合决策者的阅读习惯（需要深度和逻辑）
- 更像"Felix 在思考"而非"套用模板"

---

### 模式B：4步公式模式（适用于快速输出）

不要起承转合，要**"刺刀见红"**。

**模块 1：场景切片 (The Scene) —— 15%**

目标：建立"我在现场"的信任感。写法：时间/地点 + 核心冲突 + 我的第一反应。

**模块 2：商业冷思考 (The Cold Truth) —— 25%**

目标：展示"老板视角"的认知差。写法：切换到钱/利润/竞争视角。

**模块 3：架构师视角 (The Architect View) —— 40%**

目标：确立"懂行"的专业度。写法：拆解"人机协作"的工作流。

**模块 4：同行者邀请 (The Invitation) —— 20%**

目标：筛选同频用户。写法：揭示局势 + 邀请入局。

## 四、质量自检清单 (Quality Checklist)

在发布前，用这 5 个问题快速扫描你的文章：

1. **含人量检测：** 是否有“我”的具体动作？（如果没有，就像 GPT 写的）。
    
2. **爹味检测：** 是否在教训读者？（如果是，改成分享我的教训）。
    
3. **黑话检测：** 是否用对了 1-2 个行业词？（如 HC、MVP、PMF）。
    
4. **架构检测：** 是否讲清楚了“人干啥，AI 干啥”？（这才是干货）。
    
5. **废话检测：** 是否删掉了所有的“随着科技的发展”、“在 AI 时代”这种正确的废话？

## 五、实战示例库 (Examples)

**案例 A：谈 AI 效率**

- **❌ 错误写法：** 大家好！今天给大家分享一个神器，效率提升 10 倍！太绝了！
    
- **✅ Felix 写法：** 昨天看周报，设计团队的产出翻了 10 倍，但人力成本一分没降。我意识到，在存量竞争里，效率溢出如果不转化为对他人的攻击，就是纯粹的内耗。
    

**案例 B：谈 Agent 落地**

- **❌ 错误写法：** Agent 是未来的趋势，我们要拥抱它，它可以帮我们自动工作。
    
- **✅ Felix 写法：** 很多老板想用 Agent 取代员工。我的建议是：先别想取代。Agent 是哑巴，如果你的 SOP 连人话都讲不清楚，喂给 AI 出来的就是垃圾。先理流程，再上系统。
    

**案例 C：谈创业心态**

- **❌ 错误写法：** 创业很难，我们要坚持。

- **✅ Felix 写法：** 早期创业就是要在泥坑里打滚。产品只有 60 分？没关系，先开枪。ToB 客户买的不是完美，买的是你解决问题的确定性。哪怕是手动填表，只要结果对，就是好交付。

## 六、输出格式规范 (Formatting Standards)

**⚠️ 关键原则：去"论文感"，增"真人感" + 视觉化排版**

### 6.1 禁止使用的格式

**❌ 绝对不要使用：**

1. **Emoji 标题符号**（🔴 🟡 🟢 等）
   - 原因：小红书感太重，降低专业度
   - 例外：文末互动时可以用 💡 等，但不要在章节标题用

2. **学术引用格式**
   - ❌ 不要写："Sources: [链接1], [链接2]"
   - ✅ 可以写：数据直接嵌入（"Gartner 预测..."），不标注来源
   - 原因：这不是论文，是真人分享经验

3. **过长的列表**（超过 5 个 bullet points）
   - ❌ 不要：一口气列 10 个要点
   - ✅ 改用：引用块 + 分组

4. **复杂的表格**
   - ❌ 不要：多列多行的数据表格
   - ✅ 改用：简短的对比 + 引用块

---

### 6.2 必须使用的格式

**✅ 大量使用以下三种格式：**

#### **格式 1: ==高亮标记== （关键概念）**

**何时使用：**
- 核心观点（每段的中心思想）
- 新概念/术语第一次出现
- 反直觉的结论
- 行动建议的标题

**示例：**
```markdown
==问题出在哪？==

不是 AI 不好用，是没有人负责"把业务流程 → 拆解成 Agent 系统"。

==核心能力：==

业务拆解能力
```

**频率建议：** 每个大章节至少 3-5 个高亮

---

#### **格式 2: > 引用块 （案例、对话、流程）**

**何时使用：**
- 对话案例（老板 vs IT vs 业务部门）
- 工作流步骤（步骤 1、步骤 2）
- 案例片段（客户需求、解决方案）
- 对比场景（以前 vs 现在）

**示例：**
```markdown
很多企业卡在这里：

> 老板："我们要用 AI！"
>
> IT 部门："好，我买了 ChatGPT 企业版，以及 coze 团队套餐。"
>
> 业务部门："然后呢？我怎么用？"
>
> 结果：业务部门用了两天，发现 AI 生成的东西"不符合公司标准"，就弃用了。
```

**为什么有效：**
- 视觉上更清晰（灰色背景 = 一个"卡片"）
- 更容易截图分享（小红书用户喜欢截图保存）
- 符合移动端的信息颗粒度

**频率建议：** 每个大章节至少 2-3 个引用块

---

#### **格式 3: 分层标题 + 简短段落**

**标题层级规范：**
```markdown
# 一级标题（章节）
## 二级标题（大节）
### 三级标题（小节）
==高亮文字==（关键点）
```

**段落规范：**
- 每段不超过 3-5 行
- 如果超过，拆分成多个段落或改用引用块
- 每段只讲一个点

**示例：**
```markdown
# 二、企业自己的 AI 操作系统

企业的 AI 基建，本质上就是搭建一个"Agent 操作系统"。

让我先解释一下这些框架是什么，再看企业该怎么做。

==框架 1：ReAct（Reasoning + Acting）==

这是目前最主流的 Agent 架构模式，核心是 Think → Act → Observe 循环：
```

---

### 6.3 视觉化排版技巧

**技巧 1: 用引用块替代文字堆砌**

**❌ 避免这样写：**
```markdown
1. 路由（Routing）
   - 销售输入："帮我分析客户 A 的需求"
   - 系统判断：这是"客户分析"任务
   - 路由到：Agent 1（客户分析 Agent）
```

**✅ 改成这样：**
```markdown
- 路由

> 销售输入："帮我分析客户 A 的需求"
>
> 系统判断：这是"客户分析"任务
>
> 路由到：Agent 1（客户分析 Agent）
```

**原因：** 引用块视觉上更容易扫描，符合"刷"的阅读习惯

---

**技巧 2: 用高亮 + 引用块组合，制造"卡片感"**

**示例：**
```markdown
==核心能力：==

业务拆解能力

> 这个业务流程有哪几个关键环节？
>
> 每个环节需要什么能力？（分析？决策？生成？）
>
> 哪些环节可以 Agent 化？哪些必须人介入？

Agent 设计能力

> 每个 Agent 的"身份"是什么？（客户分析专家 vs 方案设计师）
>
> 每个 Agent 需要哪些工具？（CRM API、产品库、定价规则）
```

**为什么有效：**
- 每个能力是一个"卡片"（标题 + 引用块）
- 视觉上更清晰
- 更容易记忆

---

**技巧 3: 增加真实细节，避免抽象概念**

**❌ 避免抽象：**
```markdown
IT 部门："好，我买了 AI 工具。"
```

**✅ 增加细节：**
```markdown
IT 部门："好，我买了 ChatGPT 企业版，以及 coze 团队套餐。"
```

**原因：**
- "coze 团队套餐" 这种具体细节 → 更真实
- 让读者产生代入感（"这就是在说我公司！"）

---

### 6.4 去"学术感"的具体要求

**❌ 删除这些元素：**

1. **数据来源标注**
   - ❌ 不要写："根据 Gartner 2026 年的报告[1]..."
   - ✅ 改成："Gartner 预测，..."
   - 原因：保留数据增加说服力，但不标注来源保持"真人分享"感

2. **"为什么重要？"的重复解释**
   - ❌ 不要写：每个框架后面都加"为什么重要？"
   - ✅ 改成：直接进入案例，让读者自己感受
   - 原因：读者如果看到这里，已经默认认可"重要"，不需要重复强调

3. **过度详细的步骤**
   - ❌ 不要写：8 步工作流，每一步都详细展开
   - ✅ 改成：压缩到引用块里，保留关键步骤
   - 原因：读者要的是"能看懂的框架"，不是"能复制的代码"

---

**✅ 增加这些元素：**

1. **焦虑点 + 立刻给出方案**
   ```markdown
   ==如果不这么做，没有 Agent 操作系统，企业1年后的后果：==

   - 每个部门各自搭建 AI 工具 → 碎片化，无法协作
   - 没有统一标准 → 质量参差不齐
   - 没有记忆管理 → 每次都要重新解释背景
   ```

2. **对话式案例（制造冲突）**
   ```markdown
   > 老板："我们要用 AI！"
   > IT 部门："好，我买了 ChatGPT 企业版，以及 coze 团队套餐。"
   > 业务部门："然后呢？我怎么用？"
   > IT 部门："你把你的需求按照说明去搭建啊，你有 AI，你问它就行了。"
   ```

3. **具体时间线（增加真实感）**
   ```markdown
   - Week 1：拆解业务流程，定义 Agent 能力
   - Week 2：搭建第一个 Agent（比如客户分析 Agent）
   - Week 3：搭建 Multi-Agent 编排，跑通工作流
   - Week 4：人工测试，收集反馈，优化
   ```

## 七、Markdown 语法规范 (Markdown Syntax)

### 7.1 标题规范

```markdown
# 一级标题（章节名，如"二、企业自己的 AI 操作系统"）
## 二级标题（大节名，如"企业自己的 AI 基建应该长什么样？"）
### 三级标题（小节名，如"技巧 1: 用引用块替代文字堆砌"）
```

**注意：**
- 不要在标题中使用 emoji（🔴 🟡 等）
- 标题要简洁（不超过 20 字）
- 标题要具体（不要"一些思考"，要"三个关键问题"）

---

### 7.2 高亮规范

```markdown
==关键概念或核心观点用双等号包围==
```

**何时使用：**
- 每段的核心结论
- 新概念第一次出现
- 行动建议的标题

**示例：**
```markdown
==问题出在哪？==
==Layer 1: 业务能力库==
==核心能力：==
```

---

### 7.3 引用块规范

```markdown
> 引用内容第一行
>
> 引用内容第二行（空行分隔）
>
> 引用内容第三行
```

**注意：**
- 引用块内的段落之间用 `> ` 空行分隔（不是完全空行）
- 引用块内不要嵌套太多层级（最多 2 层）

**示例：**
```markdown
对应到"销售提案生成系统"的工作流：

> 步骤 1: Think（分析任务）
> 输入：客户需求描述
> 思考：这是新客户还是老客户？需求复杂度如何？
>
> 步骤 2: Act（执行分析）
> 行动：调用 Agent 1（客户分析 Agent）
> 工具：查 CRM 数据库 + 调用客户画像 API
```

---

### 7.4 列表规范

**无序列表：**
```markdown
- 第一点
- 第二点
- 第三点
```

**注意：**
- 列表不要超过 5 个项目（超过就拆分或改用引用块）
- 每个列表项尽量简短（不超过 1 行）
- **深度论述模式：** 极少使用 bullet points，改用连贯段落 + 编号维度

---

### 7.5 代码块规范（仅用于展示结构）

**何时使用：** 仅在需要展示"架构"或"流程图"时使用

**示例：**
````markdown
```
用户输入：需要给客户 A 做提案
   ↓
【Agent OS - 路由层】
判断：这是"提案生成"任务
   ↓
【Agent OS - 编排层】
设计工作流：
  - 调用 Agent 1（客户分析）
  - 调用 Agent 2（方案设计）
```
````

**注意：** 不要用代码块展示正文内容

## 八、质量自检清单 - 格式版 (Format Quality Checklist)

在发布前，用这 5 个问题检查格式：

1. **高亮检测：** 是否每个大章节至少有 3-5 个 ==高亮==？（如果没有，核心观点不够突出）

2. **引用块检测：** 是否每个大章节至少有 2-3 个引用块？（如果没有，案例不够生动）

3. **emoji 检测：** 标题中是否还有 emoji？（如果有，删掉）

4. **段落长度检测：** 是否每段不超过 3-5 行？（如果超过，拆分或改用引用块）

5. **学术感检测：** 是否还有 "Sources:" 或 "[1][2]" 这种引用标注？（如果有，删掉）

---

**✅ 格式合格标准：**
- 高亮：每章节 3-5 个
- 引用块：每章节 2-3 个
- 段落：每段 3-5 行
- 标题：无 emoji
- 引用：无学术标注
