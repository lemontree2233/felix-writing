# a16z说"软件的YouTube时刻来了"，但他们没告诉你企业该怎么办

刚看了 a16z 最新的 newsletter，核心观点：**软件开发正在经历一场类似视频创作的长尾爆发。**

a16z 列举了过去几周 X 上的一些现象：有人给自己搭了个 MRI 仪表盘（这在以前可是昂贵的商业软件）；有人随手搓了一个技术乐观主义风格的电影推荐器；有人直播写 App，观众看着从零到上线。

但我看完有个疑问：a16z 讲的都是软件端、个人创作者的故事。**那对于企业来说，这意味着什么？**

想了一圈，我觉得答案不是"要不要用 AI 做软件"，而是——

**在软件开发成本暴跌的时代，企业的竞争力不是"能否买到好软件"，而是"能否把内部能力变成可调用的 Skill 系统"。**

拆解三个关键问题：

1️⃣ 为什么"买 SaaS"的逻辑过时了？
2️⃣ 企业需要什么样的 AI 基建？（不是买工具，是建 Skill 系统）
3️⃣ 如何从 0 开始落地？

## 🔴 一、不要再采购 SaaS 了，它们已经跟不上 AI 的迭代速度了

**过去企业的逻辑是：**
- 通用需求 → 买成熟 SaaS（ERP、CRM）
- 独特需求 → 定制开发（昂贵、周期长）

**现在：** 人人都有"vibe coding 工具"（Cursor/Claude Code），几天就能搭出来 → 个人也能做软件。

我最近和一个跨境品牌聊，他们有个很具体的需求：**"销售团队专属的客户洞察仪表盘"**，要能自动分析客户的购买行为、标注潜在流失风险、推荐下次跟进话术。

他们问了几家 SaaS 供应商：
- **用现成 SaaS → 功能冗余 + 价格虚高**（年费几十万，但 80% 功能用不上）
- **等供应商定制 → 周期长 + 不灵活**（报价 3 个月交付，改需求要重新排期）

最后他们找了我们，我们用 AI 开发系统 + 他们的 CRM 数据，**3 天搭出了原型，2 周上线**。

这不是个例。**当 LLM 让软件开发成本暴跌时，企业会发现：很多内部需求，自己做比买更划算、更灵活。**

但问题来了：企业有这个能力吗？

## 🟡 二、竞争对手正在把内部能力"产品化"，而你还在用 Excel

a16z 文章里有个类比我很喜欢：

> 软件的发展路径和视频一样：
> 好莱坞时代（巨头垄断）→ 独立电影时代（YC 创业者）→ YouTube 时代（人人都能创作）

现在我们正在进入"软件的 YouTube 时代"。

**但这对企业意味着什么？**

举个例子。

以前，一个销售团队有一套独特的客户评分方法（Excel 表格 + 人工经验）：
- 这套方法只能口口相传，效率低
- 新人要花 3 个月才能学会
- 离职员工带走 know-how

**现在，你的竞争对手可能已经：**
- 用 LLM + 内部数据 → 2 天搭建一个自动评分工具
- **知识从"人脑"变成"可复用的软件"**
- 新人入职第一天就能用，离职员工带不走

**这就是新的护城河：能够快速将内部 know-how 转化为可运行的软件工具。**

但这背后不是买个 AI 工具，或者基于 N8N、DIFY 搭建一个工作流就行了。需要的是企业的 **Skill 系统**。

## 🟢 三、企业的 AI 基建 = Skill 系统（这是我最近的顿悟）

最近我在优化自己的 AI Agent 写作系统（详见上一篇《不会管实习生的人，管不好 AI Agent》），我突然意识到：

**企业的 AI 基建，本质上就是"企业的 Skill 系统"。**

让我解释一下什么是 Skill 系统。

### 我的 AI 写作系统是怎么工作的？

我有一个"小红书爆款创作 Skill"，它不是一个简单的 prompt，而是一个**三层结构**：

**Layer 1: Context（上下文层）**
- 文件：`个人背景.md`
- 内容：我是谁？我的价值观？我的经验和观点？
- 作用：让 AI 理解"Felix 的语言"和"Felix 的视角"

**Layer 2: Logic（逻辑层）**
- 文件：`爆款结构.md`
- 内容：写作公式、质量标准、禁止事项
- 作用：让 AI 知道"什么是好的输出"和"如何做"

**Layer 3: Example + Workflow（示例与工作流层）**
- 文件：`samples/README.md` + 5 个历史文章样本
- 内容：不同风格的案例参考 + 话题映射表
- 作用：根据话题，调用对应的风格参考；按工作流执行（采访 → 风格推荐 → 生成）

**关键设计：**
- **不是一次性加载所有知识**，而是先读索引（README.md），根据话题选择性加载对应样本
- **分层协作**：Context 定义身份，Logic 定义方法，Example 提供参考
- **可复用**：下次写文章，直接调用这个 Skill，不用重新解释

---

### 那企业的 AI 基建，为什么也是 Skill 系统？

**因为企业的每个核心能力，都应该被"Skill 化"。**

我用一个真实案例解释。

我最近帮一家服装品牌优化 AI 系统，他们的痛点是：**AI 生成的商品描述，转化率惨不忍睹。**

我看了他们的 prompt，就一句话：
**"生成这款连衣裙的商品描述。"**

我问他们的电商运营主管："如果实习生这样写，你会怎么培训他？"

她说：
- "我会先让他看我们过去卖爆的 10 个商品描述，分析为什么爆。"
- "然后我会告诉他，我们的客户是 25-35 岁的职场女性，她们在意的是'显瘦''显高''不挑身材'。"
- "最后我会给他一个模板：痛点+场景+卖点+限时优惠。"

我说："**那你为什么不把这些，变成一个 Skill？**"

## 🔵 四、企业 AI 基建的三层架构（对标 Skill 系统）

基于这个顿悟，我重新设计了他们的"商品描述生成 Skill"：

### **Layer 1: 企业知识库（Context）**

**不是简单的"知识库"，而是企业的身份定义。**

对于这家服装品牌：
- **品牌身份**：我们是谁？（性价比国货品牌，主打职场女性）
- **目标客户**：25-35 岁职场女性，身高 155-165cm，微胖身材
- **核心价值观**：不贩卖焦虑，强调"显瘦显高不挑身材"
- **历史案例**：过去 6 个月卖爆的 10 个商品描述（标注为什么爆）

**对应到我的写作 Skill：**
- `个人背景.md` = 企业的品牌定位 + 目标客户 + 价值观

**为什么重要？**
- 没有这一层，AI 生成的内容是"通用话术"，没有品牌辨识度
- 有了这一层，AI 生成的内容带着"这家公司的语言"和"这家公司的价值观"

---

### **Layer 2: 业务 SOP 库（Logic）**

**不是简单的"流程文档"，而是可执行的方法论。**

对于这家服装品牌：
- **输出模板**：痛点共鸣（1 句话）+ 场景描述（2-3 句）+ 卖点说明（3-4 个 bullet points）+ 限时优惠（CTA）
- **质量标准**：
  - ✅ 必须体现：显瘦、显高、场景（通勤/约会/聚会）
  - ❌ 禁止出现："百搭""时尚""优雅"等空泛词汇
  - ❌ 禁止：超过 20 字的长句子
- **决策框架**：
  - 如果商品是连衣裙 → 强调 A 字版型、高腰线
  - 如果商品是上衣 → 强调遮肉、显瘦
  - 如果商品是裤子 → 强调高腰、显腿长

**对应到我的写作 Skill：**
- `爆款结构.md` = 写作公式 + 质量标准 + 禁止事项

**为什么重要？**
- 没有这一层，AI 不知道"什么是好的输出"
- 有了这一层，AI 知道"按什么标准做"和"如何做决策"

---

### **Layer 3: Skill 库 + Sub-Agent 协作（Action）**

**这是企业 AI 基建最被低估的一层。**

很多企业以为"AI 基建"就是买个 ChatGPT 企业版，或者搭个 RAG 知识库。

**错了。**

**真正的 AI 基建，是把业务能力拆解成可调用的 Skill，并设计 Sub-Agent 之间的协作关系。**

让我继续用服装品牌的例子。

**他们的完整业务流程是：**
1. 商品上新 → 需要生成商品描述
2. 描述生成后 → 需要审核（是否符合品牌调性？是否有敏感词？）
3. 审核通过 → 发布到电商平台
4. 发布后 → 监控数据（点击率、转化率）
5. 如果转化率低 → 需要优化描述

**如果把这个流程"Skill 化"：**

```
主控 Agent（业务协调者）
   ↓
调用 Skill 1: "商品描述生成 Skill"
   - 输入：商品信息（款式、面料、场景）
   - 输出：3 个版本的商品描述
   ↓
调用 Skill 2: "内容审核 Skill"
   - 输入：3 个描述 + 品牌规范库
   - 输出：审核结果（通过/拒绝，理由）
   ↓
调用 Skill 3: "数据监控 Skill"
   - 输入：商品 ID + 电商平台 API
   - 输出：点击率、转化率、用户评论
   ↓
调用 Skill 4: "内容优化 Skill"（如果转化率低）
   - 输入：原描述 + 数据反馈
   - 输出：优化建议 + 新版本描述
```

**这就是 Sub-Agent 协作。**

**对应到我的写作 Skill：**
- 我的写作工作流：采访 → 风格推荐（查索引 README.md）→ 加载对应样本 → 生成文章
- 每个步骤都是一个可调用的模块

**为什么重要？**
- 这一层决定了"AI 能不能真正执行业务"，而不只是"给建议"
- 没有这一层，AI 只是"顾问"；有了这一层，AI 是"执行者"

---

### **关键设计原则：按需加载 + 索引路由**

回到我的写作 Skill，最关键的设计是：

**不是一次性加载所有样本文章（那样会消耗大量 Token），而是：**
1. 先加载索引（`README.md`），里面有"话题 → 风格"的映射表
2. 根据用户的话题，推荐 2-3 个风格选项
3. 用户选择后，只加载 1 个样本文章
4. 按照这个样本的风格生成新文章

**对应到企业：**
- 不是把所有业务知识一股脑扔给 AI
- 而是设计一个**"业务场景 → Skill"的路由系统**

**示例：销售部的 Skill 路由系统**

```
用户输入：需要分析客户 A 的购买意向
   ↓
路由系统判断：这是"客户分析"场景
   ↓
调用 Skill：客户分析 Skill
   - 加载：客户画像库 + 历史成交案例
   - 执行：分析客户行为 + 预测流失风险 + 推荐话术
   ↓
输出：客户分析报告 + 下次跟进建议
```

**为什么这样设计？**
- **效率**：不需要每次都加载所有业务知识
- **精准**：根据场景加载对应的 Skill，输出更准确
- **可维护**：每个 Skill 独立，更新一个不影响其他

## 🟣 五、这不是技术问题，是组织能力问题

很多企业卡在这里：

**老板：** "我们要用 AI！"
**IT 部门：** "好，我买了 ChatGPT 企业版。"
**业务部门：** "然后呢？我怎么用？"
**IT 部门：** "自己琢磨啊，这是 AI，你问它就行了。"

**结果：** 业务部门用了两天，发现 AI 生成的东西"不符合公司标准"，就弃用了。

**问题出在哪？**

**不是 AI 不好用，是没有人负责"把业务需求 → 转化为 Skill 系统"。**

**这需要一个新角色：Skill 架构师（Skill Architect）。**

这个角色：
- **不是传统程序员**（写底层代码）
- **不是产品经理**（只画原型）
- **而是：能够把业务能力拆解成 Skill，并设计 Sub-Agent 协作流程的人**

**核心能力：**
1. **理解业务痛点**（业务敏感度）
   - 销售部最耗时的是什么？客户分析？还是写提案？
   - 运营部最头疼的是什么？数据清洗？还是报告生成？

2. **设计 Skill 架构**（系统思维）
   - 把业务能力拆解成可复用的 Skill
   - 设计 Skill 之间的调用关系（谁先谁后？谁调用谁？）

3. **用 LLM 快速实现**（提示工程 + 工具链）
   - 不需要从零写代码
   - 用 LLM + 低代码工具快速搭建 Skill

**这个角色的价值，不是"写代码"，而是"把内部知识产品化"。**

## 🔷 六、给企业管理者的三个建议

如果你认同"软件长尾化会改变企业竞争力"，那我建议你从这三件事开始：

### 1. 别急着上 AI，先问自己"我有没有 SOP"

在引入 AI 之前，先问：
- 如果这件事交给实习生做，我会给他什么培训材料？
- 我会给他什么参考案例？
- 我会设计什么质量标准？

**如果你连这些都没有，先别急着上 AI。因为你缺的不是工具，是管理框架。**

### 2. 从一个高频场景开始，搭建第一个 Skill

不要一上来就想"全公司 AI 化"。

**从一个具体的、高频的、有明确标准的场景开始：**
- 比如：客户邮件回复（有标准话术）
- 比如：周报生成（有固定模板）
- 比如：商品描述撰写（有历史案例）

**用 2 周时间，搭建一个完整的 Skill：**
- Layer 1: 整理业务知识（客户画像、历史案例）
- Layer 2: 定义 SOP（输出模板、质量标准）
- Layer 3: 设计工作流（人在哪一步介入？AI 在哪一步执行？）

**跑通了，再复制到其他场景。跑不通，说明基建还没准备好。**

### 3. 找一个"会管人"的人，让他去做 Skill 架构师

不要让 IT 部门主导 AI 落地。IT 部门懂技术，但不懂业务。

**找一个业务部门的人（最好是那种"会带实习生"的人），让他负责：**
- 整理业务知识库（过去的成功案例、话术库、标准模板）
- 设计 Skill 工作流（什么时候用 AI、什么时候人工介入）
- 持续优化（定期 review AI 输出质量、收集翻车案例）

**这不是技术问题，是管理问题。**
